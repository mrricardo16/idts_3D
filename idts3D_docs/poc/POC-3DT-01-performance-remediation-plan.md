# POC-3DT-01 性能修复方案冻结与实施拆分

> 冻结日期：2026-07-21
>
> 依据：`2759345 Add POC performance diagnosis scenario and probe`、[性能诊断](POC-3DT-01-performance-diagnosis.md)及其原始 JSON。
> 本文是后续独立修复任务的方案基线，不实施模型处理、缓存头改造、代码改造或依赖安装。

## 1. 当前状态与结论边界

```text
POC-3DT-01：Fail
MVP-10A-01～MVP-10A-05：Blocked
解锁建议：Do not recommend unlock MVP-10A-01
性能诊断：Diagnosis complete
```

现有平均 FPS 已高于 `>= 45` 的暂定门槛，但这不能覆盖真实 GLB 场景约 `29.94` 的 P5 FPS、约 `11.99～14.99` 的最低 FPS、长任务以及每次 reload 约 `279.61 MiB` 的实际传输。本文不改变任何正式结论或门槛。

## 2. 已确认的性能归因

| 维度 | 已确认事实 | 结论 |
| --- | --- | --- |
| 网络 | 开发、预览的 `lifter.glb` 都是 `Cache-Control: no-cache`；冷、暖缓存均约 293,193,197～293,196,341 B CDP `encodedDataLength` | 暖缓存未形成有效 GLB 复用；缓存是独立问题线。 |
| 绘制 | 真实 GLB 单独约 1,087 calls / 8,526,166 runtime triangles；加厂房后为 1,095 / 8,526,262 | 真实 GLB 的碎片化绘制是 draw-call 主因。 |
| 几何 | 源 GLB 约 854 万三角形；真实 GLB 场景 JS Heap 约 297～313 MiB | 高几何量是 P5 与长任务的主要候选根因，需优化副本验证。 |
| 厂房 Tiles | 仅约 10 calls / 110 triangles；增加到真实 GLB 上仅增加约 8 calls / 96 triangles | 不是主要瓶颈。 |
| 诊断面板 | 降低统计刷新频率没有改善 P5 | 不是主要瓶颈。 |
| 渲染生命周期 | 已取证为单 Canvas、单 Renderer、单 rAF、每帧一次 Controls 更新 | 不是当前根因。 |

缓存修复不能直接降低 draw calls、三角形、P5 低帧或 GPU 压力；反过来，网格优化也不能消除 293 MiB 的重复响应体传输。后续工作必须分别测量两条线。

### 3.2 碎片化与重复结构

| 检查 | 结果 | 可执行含义 |
| --- | --- | --- |
| 小 primitive | 21 个不超过 100 triangles；634 个不超过 1,000 triangles（1,123 总数） | CAD 导出颗粒度很细；仅小件本身不足以解释全部成本，但合并空间显著。 |

## 4. 节点语义、拾取和运动风险

GLB 是静态导出快照：没有 skin、animation 或明确的 `lifter-platform` 节点。技术上 1,123 个 mesh 节点均未被 animation/skin 引用；这**不能**证明它们在业务上都是静态件，也不能识别可动部件。

| 分类 | 可确认范围 | 数量 / 状态 | 后续限制 |
| --- | --- | --- | --- |
| A 静态主体 | 仅“未被 skin/animation 驱动”的技术候选 | 1,123 个技术静态候选，未获业务确认 | 不可直接用于 mesh 合并白名单。 |
| B 可运动部件 | 需要建模源语义、设备清单或项目方确认 | 0 个已确认 | 禁止根据 `NAUO` 编号、空间位置或外观猜测。 |
| C 可拾取业务对象 | 当前 POC 可拾取到 scene object 名 | `NAUO682` 曾被 E2E 拾取，但无业务语义 | 不可将其视作稳定 `deviceId` 或可动平台。 |
| D 仅装饰细节 | 需要业务/建模方标注 | 0 个已确认 | 禁止按三角形数量直接删除。 |
| E 重复标准件 | binary geometry 完全一致的候选 | 137 组 / 892 个额外项 | 可做实例化 POC；需保持逐实例可选与可追溯。 |
| F 不可确认 | `NAUO***` 自动名及未标注部件 | 至少 1,188 个 | 默认保护，不能合并、简化、删除或归类为可动。 |

`NAUO***` 看起来是 CAD 导出自动名，当前没有证据证明重导出后编号稳定；因此所有后续任务把它当作“当前构建内技术标识”，不能当作跨版本业务主键。推荐的长期身份保护是：

1. 建模/资产生产阶段提供稳定的 `sourceObjectId` 与业务节点名，不以 `NAUO` 替代。
2. 在优化产物旁新增版本化、可审计的 `object-index.json`：每个 `sourceObjectId` 映射原节点名、优化 node/mesh、实例 `instanceId`（如有）、face range（如有）、可动标记和 SHA-256。
3. 合并后使用 `geometryGroup/faceRange → sourceObjectId`；实例化后使用 `instanceId → sourceObjectId`；运行期写入只读 `userData.sourceObjectIds`，但 sidecar 是权威来源。
4. 运动部件必须先从静态合并集合排除，保留独立 node/transform。优化 manifest 记录原始 SHA、优化 SHA、映射版本和可回退原始 URL。

## 5. 静态资源托管链路审计与缓存方案

### 5.1 已知与待确认的托管层

| 环境 | 已知事实 | Cache-Control 现状 | 结论 |
| --- | --- | --- | --- |
| Vite 开发服务器 | POC 从 `/models/lifter.glb` 提供 | `no-cache`，有 ETag/Last-Modified/Content-Length | 暖 reload 仍完整传输，不是生产方案。 |
| Vite preview | 诊断结果与开发一致 | `no-cache`，有 ETag/Last-Modified/Content-Length | 仅测量对照，不得假设为生产。 |
| ASP.NET Core | 仓库 API 能通过 `UseStaticFiles` 将 `FileStorage.RootPath` 映射为 `/assets`，并登记 GLB content type | 当前源码没有为此路由冻结缓存策略 | 是“可用源站实现”，不是生产部署证据。 |
| Nginx / CDN / 对象存储 / 文件服务器 | 本次在允许范围内未发现生产拓扑、反向代理或对象存储配置证据 | 未确认 | **正式生产托管链路待用户或项目方确认。** |

在正式拓扑确认前，不得把 Vite 或 ASP.NET Core 的默认响应行为写成生产承诺。单一被指定的静态资源权威层应负责：`Cache-Control`、`ETag`、`Last-Modified`、`Content-Length`、`Accept-Ranges`；若 CDN/代理进行压缩，则还要对 `Content-Encoding`、`Vary: Accept-Encoding` 和实体校验器一致性负责。GLB 的二进制压缩收益应先用实际文件测量；不要无条件开启 HTTP 压缩而造成 CPU 成本或 range 语义问题。

### 5.2 缓存路线比较

| 方案 | 网络收益 | 部署/回滚 | 适用性 | 本轮建议 |
| --- | --- | --- | --- | --- |
| A. 版本化 URL + 长缓存 | 内容不变时暖缓存可接近 0 响应体；例如 `/assets/models/lifter.<hash>.glb` + `public, max-age=31536000, immutable` | 发布新 hash 即更新；旧 URL 保留可回滚 | 生产首选，但需资产 manifest 与托管层确认 | P1 首选。 |
| B. 短缓存 + 条件请求 | 可能 304，降低响应体但仍有往返和验证成本 | 简单；版本更新无需改 URL | 开发、频繁变更资产或无法版本化时的保守方案 | 不作为 293 MiB 正式资产首选。 |
| C. Service Worker / Cache Storage | 可离线、跨刷新复用 | 需版本清理、失效、配额和调试策略 | 当前没有资产版本/淘汰治理 | 不在 P0～P3；仅在 A 无法满足离线需求后单独评审。 |
| D. 应用内 ArrayBuffer 缓存 | 只减少同页重复加载 | 跨刷新无收益，约 293 MiB 内存风险 | 不适合主要方案 | 仅诊断或一次会话的局部优化，不进入正式路线。 |

候选验收以 CDP `Network.loadingFinished.encodedDataLength` 为主口径，不能以 `transferSize`、解码体积、磁盘文件大小或页面累计资源大小替代。

## 6. 模型优化路线比较与决策矩阵

| 方案 | 网络收益 | FPS / calls 收益 | 节点、拾取风险 | 成本 / 回滚 | 推荐阶段 |
| --- | --- | --- | --- | --- | --- |
| 仅缓存头 | 高（暖加载） | 无 | 无 | 低；换回旧头/URL | P1 |
| 属性清理、二进制去重、Meshopt / Draco | 中到高（视资产） | 对 calls 基本无直接收益；解码成本需测 | 低到中；必须比较 node/名称/映射 | 中；新副本可删除回退 | P2 |
| 按材质合并明确静态 mesh | 小到中 | 高潜力 | 高：可能丢失 node/face 业务身份 | 中高；只使用新副本 | P3 |
| 重复构件实例化 | 小到中 | 高潜力 | 高：要维护 instanceId 映射 | 中高；仅对 137 组候选验证 | P3 |
| 三角形简化 / LOD | 中（多级加载） | 中到高，取决于视距 | 中到高：工业细节、测量和拾取风险 | 高；需要视觉误差批准 | P4（必要时） |
| 静态/运动拆分 | 视加载策略而定 | 可使静态优化不影响运动件 | 高但可控；依赖业务标注 | 高；需要 manifest/index | P3 前置或并行 |
| 静态主体 3D Tiles | 视锥/LOD 情况下可降首次可见成本 | 对单台 2.35×3.14×23.08 m 设备未必划算 | 高；静态和运动资产需分离 | 高；转换/维护链复杂 | P4，非 POC 临时硬编码 |

不推荐把文件压缩描述成 FPS 修复，也不推荐在不具备业务节点映射时直接合并或简化。当前源 GLB 无纹理，纹理压缩不进入首轮优化。

## 7. 冻结的分阶段路线

| 阶段 | 目的 | 进入条件 | 停止 / 升级条件 |
| --- | --- | --- | --- |
| P0 测量与资产保护 | 原始 GLB 只读保留；记录 SHA-256、节点/拾取/视觉基线；建立优化副本对照规则 | 用户批准独立后续任务 | 无可靠业务身份或基线不稳定时停止。 |
| P1 缓存修复 | 确认生产静态源；版本化 URL、immutable 长缓存、ETag/Last-Modified/Range；CDP 验证暖传输 | 托管层和发布方式获确认 | 暖传输未达到候选门槛，先修托管而非进入模型优化。 |
| P2 低风险压缩副本 | 保持 hierarchy、名称和映射；只生成 `optimized-v1` 新副本并比较 bytes/加载/视觉 | P0 资产基线与工具授权完成 | node/name/拾取回归即丢弃副本。 |
| P3 合并、实例化与运动保护 | 仅对项目方确认的静态节点执行；完善 object index | B 类运动节点、C 类业务对象、E 类重复件的名单完成 | 任何身份、拾取、运动绑定不一致即不扩大范围。 |
| P4 LOD 或静态主体 3D Tiles | 当 P2/P3 后稳定性仍不达标时，建立生产资产规范和 manifest | P3 有可审计映射但指标仍不足 | 不在 POC 中硬编码；需单独审批。 |

P0～P4 只定义推荐顺序，不表示已批准实现。

## 11. 风险、回退与用户待确认事项

**全局回退：** 原始 `lifter.glb` 永久只读；所有优化产物新文件名、独立 SHA、独立 manifest；POC 必须能显式选择原始和优化 URL 做 A/B 对照。任何回归只切回原始 URL，不覆盖原文件。每次优化前后保存同一相机视觉、拾取、生命周期、CDP 与 30 秒性能对照。

用户或项目方须先确认：

1. 是否允许新增受控模型处理工具和依赖；是否有可用的 CAD、FBX、STEP 或 Blender 源文件。
2. 是否允许生成优化后的新 GLB 副本，以及是否允许单独性能修复分支/任务。
3. `NAUO` 是否具有业务意义；是否必须保留每个 NAUO 的独立拾取能力；谁能给出稳定 `sourceObjectId`。
4. 建模人员能否明确标记运动部件，以及是否接受静态主体/运动部件拆分。
5. 是否允许对**已确认**静态节点做 mesh 合并和实例化；是否允许静态主体转 3D Tiles。
6. 生产模型最终由 Nginx、ASP.NET Core、对象存储、CDN 或其他层托管；谁拥有缓存头和发布回滚权限。
7. 是否批准 P5 FPS、暖缓存传输、draw-call 目标和 Long Task 的候选门槛。
