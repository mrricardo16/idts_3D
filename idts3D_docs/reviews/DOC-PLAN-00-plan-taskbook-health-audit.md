> 审计类型：全量文档审计；仅新增本报告。

> 审计边界：未修改既有文档、业务代码、配置、模型或数据库；未执行 POC、构建、测试、依赖安装、`git add`、commit 或 push。

## 1. 审计结论

**最终结论：Conditional Pass。**

当前文档已经具备一条可审查的 GLB + 3D Tiles 混合场景路线：资源职责、POC 三态门禁、`baseLayers + devices` 方向、坐标/生命周期/性能/回退设计和 MVP-10A 子任务链均有明确来源。它可以作为**设计与文档审核基线**继续使用。

但它目前不能作为“无需再核对即可执行下一项实现任务”的正式任务书：根 README、后端 README、架构基线与 MVP-08 任务卡对已实现范围和测试/CI 状态作出了相互矛盾的陈述，直接影响前置条件判断；MVP-10A 子卡又缺少与既有 MVP 卡同等粒度的实施边界和验证细节。因此，应先完成本报告 P0 的文档事实校准，随后逐项授权实施。

**事实**：本审计读取的现行总纲和任务索引均写明 POC 不阻塞 MVP-09/MVP-10、直接阻塞 MVP-10A，且 MVP-10A-01～05 均为 Blocked。
**推断**：混合方案的方向性决策已足以支撑 POC 准备，但项目进度事实不能可靠地自动推导出下一张可执行实现卡。
**建议**：先校准实现状态的唯一事实源，再进行 POC 的单独执行授权或任何实现授权。

## 5. 战略与目标审计

**事实**：总纲、任务索引和 ADR 均将 3D Tiles 定义为厂房、道路、固定设施等静态底座，将 GLB 定义为动态设备、Object Tree、Movable Part、Motion Target、Edit/Monitor、worldZ、拾取和告警的承担者。`tilesets: []` 被明确标为兼容占位，正式方向是 `baseLayers + devices`。

**事实**：总纲明确排除完整 CAD/IFC 自动转换、生产切片平台、完整 3D Tiles 资产管理、未授权真实数据和最终现场性能承诺；MVP-15 亦未把 GLB 转换日志扩大为 Tileset 工具链。

**审计判断**：战略边界健全，不存在“以 3D Tiles 取代动态 GLB 业务对象”的当前冲突。场景与数据模型中的 TBD 不是缺陷：它们被明确限定为 POC/现场资料后、MVP-10A-01 审核冻结前不得实施的事项。

## 6. 计划结构审计

**事实**：当前总纲的唯一顺序为“文档审核 → POC → 结果审核 → 用户批准 → MVP-09/10 → MVP-10A-01～05 → MVP-11～16”；POC 不阻塞 MVP-09/MVP-10 和无关纯后端工作，但阻塞全部 MVP-10A 子任务。

**发现 PLAN-P1-01**：MVP-11 的任务卡仅以前置 MVP-03～10 当前契约稳定为条件，MVP-15 仅以前置 MVP-02、MVP-03 和 conversion-jobs 契约为条件；两卡均是当前 GLB/API 或 GLB job 范围。任务索引却把 MVP-11～16 合并排在全部 MVP-10A 子任务之后，未区分“正式混合验收依赖”与“当前 GLB 基线可并行”。

**推断**：单人执行者可能把索引顺序误解为对 MVP-11 或 MVP-15 的硬阻塞，违背“POC 不阻塞无关任务”的治理意图，或反过来绕过正式混合任务实施下游页面。

**建议**：在索引和总纲中增加每张 MVP-11～16 的显式依赖边，以及 `可并行 / 必须阻塞 / 仅正式混合验收阻塞` 三种状态；不要仅依赖段落文字推断。

## 7. 任务卡完整度审计

**事实**：MVP-00～MVP-10 原有任务卡通常包含目标、前置、影响范围、禁止范围、后端、前端、数据库、API、前后端对应、步骤、验收、回归、风险、回滚和 Codex 提示词等 15 个左右结构段。

**事实**：MVP-10A-01～05 各自仅有状态行和一个压缩段落，虽包含目标、前置、验收、回滚与提示词关键词，但没有独立的文件影响范围、跨端字段表、验证命令/证据格式、风险责任人和到下一子卡的退出条件。MVP-11～16 已有 4～5 节结构，但也没有恢复到旧卡的完整粒度。

**发现 PLAN-P1-02**：MVP-10A 总卡声明“每张子卡必须独立授权、验收、回归和回滚”，但子卡本身不足以让单人实施者在不重新补充计划的情况下逐项执行。

**建议**：将新的实现卡统一到一个最小模板；至少固定“目标、前置事实、允许/禁止文件范围、API/DTO/数据库影响、步骤、验收证据、回归、风险、可操作回滚、下一任务入口、Codex 提示词”。这只是文档修订建议，不是对任何子卡的实施授权。

## 8. 前后端与数据库一致性审计

**事实**：`api-contracts/scenes.md` 的当前响应仍为 `devices` 加空 `tilesets`；Scene Resource Manifest 设计明确为草案，禁止在 MVP-10A 审核前变更 API、DTO、TypeScript 或数据库。实体映射也明确禁止用 `device_instance` 或 `device_model_binding` 表示静态厂区底座。

**发现 PLAN-P0-01：实现状态事实相互冲突。**

- `README.md` 说后端“当前已完成 MVP-01 至 MVP-07”，且把 Motion Target API、自动化测试和 CI 列为未实现。
- `idts3D_api/README.md` 同样说只完成 MVP-01 至 MVP-07，并称 Movable Part、Motion Target、自动化测试和 CI 尚未完成。
- `mvp-tasks/MVP-08-motion-target-api.md` 声明 Implementation Status 与 Automated Test Status 为 Completed、Delivery Status 为 Ready after CI。
- `architecture/project-architecture-baseline.md` 记录 ARCH-03A～03D 的测试/CI 基线，并在只读目录核验中可见 `MotionTargetsController` 及对应 Application/Contracts/Infrastructure 入口。

**影响**：后续任务无法可靠判定 MVP-08、可动部件、自动化测试/CI 是否可作为前置事实；也无法可靠安排真实 PostgreSQL 验证债务。

**建议**：建立一份被 README、API README、架构基线和任务索引共同引用的“实现状态账本”，按“已实现、已自动验证、待真实环境验证、计划中”分栏；以当前 Git 提交和可复核证据更新三个冲突入口。不得以目录存在替代运行验证。

## 9. 3D 场景设计审计

**事实**：ADR 状态为 `Accepted for design baseline`，坐标、混合场景、生命周期、Manifest、性能和回退文档均标为 `Draft for review` 或 POC 审核状态，且全部声明不授权编码。

**事实**：坐标规范将原点、轴、Y-up/Z-up、旋转、绝对缩放、高度基准和最终容差标为 TBD；POC 要求三标定点、显式变换及证据记录。生命周期文档覆盖初始化、加载、更新、隐藏、卸载、销毁、请求取消、缓存、场景切换、页面退出、错误恢复和 WebGL 释放。

**审计判断**：3D 设计文档没有伪造现场坐标或性能实测值，职责划分与回退原则一致。当前不足不是设计缺失，而是它们尚未经过用户审核和 POC 证据闭环；因此 MVP-10A 维持 Blocked 是正确状态。

## 10. 测试与验收审计

**事实**：POC 测试计划规定了许可、阶段 A/B、Mandatory/Optional、三个标定点、GLB 拾取/Object Tree/worldZ、三种 Tiles 失败、连续 10 次进出、请求取消、纯 GLB 对照及冷暖缓存轮次；结果模板要求环境、硬件、数据许可、性能、内存、请求、误差、问题责任人和三态结论。

**事实**：E2E 计划要求正式 `baseLayers` 和 GLB `devices`、坐标校验、Edit/Monitor、worldZ、告警/高亮、失败回退和资源释放，并明确 POC、空 `tilesets` 或仅构建成功不能替代正式验收。

**发现 PLAN-P1-03**：项目文档对“自动化测试和 CI 是否已建立”存在 P0 所述矛盾，且 MVP-08 已把真实 PostgreSQL CRUD、事务和行锁验证登记为 `MVP-08-VERIFY`。因此，验收体系的**设计**较完整，但当前可用验证证据的状态不可靠。

## 11. 项目治理审计

**事实**：根规则与开发规则定义了文档/MVP/验收阶段、单任务授权、必读顺序、前后端契约同步、影响/风险/验证/回滚输出，以及 main 交付前的远端同步、干净工作区和验证要求。

**事实**：本次开始时 `git status --short` 为空、当前分支为 `main`、最新提交为 `5a9a2c5`；没有已暂存或删除文件。

**审计判断**：治理门禁明确，特别适合单人 + Codex 的最小范围实施。缺口在于“当前状态事实源”并未纳入同一门禁，导致治理流程能保证执行姿势，却不能保证选中的前置事实正确。

## 12. 单人 + Codex 可执行性审计

**事实**：总纲要求一次只执行一个明确任务，POC 和每张实现卡都要求单独授权；MVP-10A 被拆成五张顺序子卡，降低了一次性跨端修改的范围。

**推断**：POC 及旧版 MVP-00～10 卡对单人可执行性较好；MVP-10A 子卡没有列出精确输入、文件边界、验收命令和证据提交位置，执行者仍需在实施前临时补计划，增加偏离任务范围的风险。

**建议**：在每张新卡中增加一个可复制的“开工检查清单”和“完成证据清单”；同时把外部批准（POC Go/Conditional Go、设计评审）记录为带日期/批准人的门禁事实。

## 16. P0 问题

- `idts3D_api/README.md`：首段、`当前已实现的业务能力` 段和“当前尚未包含”段。

**事实**：上述文件对 MVP-08、Movable Part、Motion Target、自动化测试和 CI 的已实现/未实现描述无法同时为真；本次只读结构核验还确认 Motion Target 与 Movable Part 的后端命名入口存在。

**必须先做的文档动作**：核对当前提交和实际验证证据后，统一状态语义；把“代码存在”“自动化验证完成”“真实 PostgreSQL/浏览器/WebGL 未验证”分开记录。未关闭前，不得依据 README 单独判定任何后续任务已满足前置。

## 17. P1 问题

1. **PLAN-P1-01：依赖表达不足。** 任务索引的默认串行顺序不能替代每张任务卡的硬依赖；尤其需要明确 MVP-11 与 MVP-15 是否可在 POC/10A 期间以 GLB 范围单独执行。
2. **PLAN-P1-02：MVP-10A 子卡不可独立落地。** 子卡保留了正确方向，但未达到既有卡的实施颗粒度，无法满足父卡“独立授权、验收、回归和回滚”的要求。
3. **PLAN-P1-03：验证能力状态未与债务闭环对齐。** `MVP-08-VERIFY` 与架构债务已指出真实 PostgreSQL/浏览器/WebGL 等边界，但 README 仍笼统称没有自动化测试/CI；应以分层验证状态统一呈现。
4. **POC 批准记录缺失。** POC 处于 `Ready for user execution approval` 是正确状态，但尚没有用户批准、可用许可数据或结果报告；它不能被当作已完成前置。

## 18. P2 问题

1. 任务索引将 MVP-00～08 与 MVP-11～16 汇总链接，逐卡可发现性较低；不影响链接有效性，但增加人工定位成本。
2. 多份历史方案、需求补充和审计报告仍保留长期愿景与历史结论。现行文档已大体通过“唯一执行基线/历史报告”标签消除歧义；建议持续在新文档顶部声明其属于现状、计划、草案还是历史证据。
3. `TBD` 的批准、关闭人与日期字段尚未统一。当前不应为此猜测现场值，但可建立待决项登记格式。
